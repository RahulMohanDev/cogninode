cogninode beta — architecture map

The whole app, on one page.

cogninode is a tree-shaped AI chat app that runs entirely in the browser. There is no server, no auth library, no state manager, no HTTP client. Just React 18 + Dexie 4 + a direct fetch to OpenRouter. This page documents every component, every hook, every storage location, and every data flow against the actual source committed to main.

1 · overview

What cogninode is

A fully client-side React app: chat is shaped as a tree of branched nodes; every node holds messages; the “path” from root to the current node is what gets sent to the model. All persistence lives in IndexedDB (via Dexie). The only network traffic is the user’s own OpenRouter API key calling POST /api/v1/chat/completions with SSE streaming.

5 · hooks

Four small hooks. That’s the whole state layer.

cogninode does not use Redux or any global store. Three of the four hooks are thin useLiveQuery wrappers around Dexie tables; the fourth (useStream) is the only piece of stateful orchestration in the app.

useSettingssrc/hooks/useSettings.ts

Persistent preferences hook. Loads cogninode_api_key + cogninode_prefs from localStorage on mount. Exposes setters that write through. Theme has its own dedicated key (cogninode_theme) so the pre-paint bootstrap in index.html can read it before React mounts.

function useSettings(): { apiKey: string; setApiKey: (key: string) => void; clearApiKey: () => void; prefs: Prefs; setPref: <K extends keyof Prefs>(key: K, value: Prefs[K]) => void; setTheme: (mode: ThemeMode) => void; }

Default prefs: { defaultModelId: "flash", branchMode: "follow", customModels: [], theme: "dark" }.

Consumers: ApiKeyGate, Composer, Sidebar, Message, ChatApp, SettingsModal.

Also exports a standalone applyTheme(mode) function so non-React call sites (e.g. main.tsx bootstrap) can flip the theme imperatively.

useStreamsrc/hooks/useStream.ts

Owns the streaming state machine: idle | streaming | error. On send() it (1) persists the user message to Dexie, (2) auto-titles the chat if it’s still “New chat”, (3) builds the path context via buildPathMessages, (4) fires streamMessage from lib/stream.ts, (5) accumulates content + reasoning chunks into local state, (6) on onDone persists the assistant message with token counts + cost, (7) on onError deletes the user message it persisted so the UI doesn’t lie about what was sent.

function useStream(chatId: string, nodeId: string): { state: "idle" | "streaming" | "error"; streamingText: string; streamingReasoning: string; error: string | null; send: (params) => Promise<void>; cancel: () => void; }

Internal state: state, streamingText, streamingReasoning, error, abortRef (AbortController).

Side effects: writes to db.messages (user + assistant), updates db.chats (title + updatedAt), updates db.nodes (root label).

Consumers: ChatApp only.

Bug fix this hook embodies: the previous implementation did pathMessages.slice(0, -1) which silently dropped the user’s current question. That meant first-turn messages were sent with only the system prompt and got “State your need clearly” replies. The current code persists the user message before calling buildPathMessages so the message is naturally the last item in the request body.
useCostEstimatesrc/hooks/useCostEstimate.ts

Reactive composer cost estimate. Reads all nodes in the chat via useLiveQuery, walks parents to compute depth, then assumes ~400 chars per ancestor node and feeds the result into estimateCostUsd.

function useCostEstimate( composerText: string, nodeId: string, chatId: string, modelId: string, ): number

Consumer: Composer (drives the cost pill).

// src/lib/db.ts — lines 139–209
export async function buildPathMessages(
  chatId: string,
  nodeId: string,
): Promise<Array<{ role: "user" | "assistant"; content: string | unknown[] }>> {
  const allNodes = await db.nodes.where("chatId").equals(chatId).toArray();
  const nodeMap  = new Map(allNodes.map(n => [n._id, n]));

  // Walk to root
  const path: Node[] = [];
  let currentId: string | null = nodeId;
  while (currentId) {
    const node = nodeMap.get(currentId);
    if (!node) break;
    path.unshift(node);
    currentId = node.parentId;
  }

  // Collect messages in path order
  const result: Array<{ role: "user" | "assistant"; content: string | unknown[] }> = [];

  for (const node of path) {
    const msgs = await db.messages
      .where("nodeId").equals(node._id)
      .sortBy("createdAt");

    for (const msg of msgs) {
      if (msg.role === "user" && msg.fileIds?.length) {
        const files = await db.files
          .where("_id").anyOf(msg.fileIds)
          .toArray();

        const images    = files.filter(f => f.kind === "image");
        const nonImages = files.filter(f => f.kind !== "image");

        const textChunks: string[] = [];
        for (const file of nonImages) {
          if (file.kind === "pdf") {
            textChunks.push(`<document name="${file.name}">\n${file.content}\n</document>`);
          } else if (file.kind === "code") {
            const ext = file.name.split(".").pop() ?? "";
            textChunks.push("```" + ext + "\n" + file.content + "\n```");
          } else {
            textChunks.push(`<file name="${file.name}">\n${file.content}\n</file>`);
          }
        }
        if (msg.content) textChunks.push(msg.content);
        const textContent = textChunks.join("\n\n");

        if (images.length === 0) {
          result.push({ role: "user", content: textContent });
        } else {
          const parts: unknown[] = [{ type: "text", text: textContent }];
          for (const img of images) {
            parts.push({ type: "image_url", image_url: { url: img.content } });
          }
          result.push({ role: "user", content: parts });
        }
      } else {
        result.push({ role: msg.role, content: msg.content });
      }
    }
  }

  return result;
}

lib/stream.ts184 lines

The OpenRouter SSE caller. Uses eventsource-parser so partial lines, CRLF newlines, and OpenRouter’s : OPENROUTER PROCESSING heartbeat comments don’t corrupt the stream. Splits delta.content from delta.reasoning (or reasoning_content for upstream providers that leak the original key) and emits each through a separate callback.

Exports: StreamEvent, streamMessage.

Importers: hooks/useStream.

Handles HTTP-level errors with friendly messages: 401 → "Invalid API key. Check Settings."; 402 → "Insufficient OpenRouter credits."; 429 → "Rate limited. Wait a moment." Other statuses surface OpenRouter’s parsed error.message or fall back to "HTTP <status>".

When finish_reason === "length", it appends a final chunk so the user knows the cap was hit: "_(response hit max-tokens limit — bump the cap in src/lib/stream.ts if you need longer answers)_"

lib/files.ts84 lines

File attachments. inferKind(file) classifies as image / pdf / code / file. readFileContent(file, kind) base64-encodes images (data URLs), extracts text from PDFs via a dynamic import("pdfjs-dist") + a worker bundled by Vite (?url import), or just reads file.text() otherwise. storeFile(file) persists the result into db.files and returns a lightweight ProcessedFile chip.

Exports: inferKind, readFileContent, ProcessedFile, storeFile.

Importers: components/chat/Composer.

Composer is no longer polluted. Earlier the file’s text was being dumped into the composer textarea. Now content is stored in db.files, and injection into the prompt body happens at buildPathMessages time — so the user sees a chip in their message, the model sees the document.
lib/export.ts88 lines

JSON backup. exportAllChats() dumps every row from all five tables into a single JSON blob and triggers a download. importFromJson(file) reads the file, filters to chats whose _id is not already in the DB, then bulk-adds only the nodes / messages / reflections / files that belong to those new chats.

Exports: EXPORT_VERSION (= 1), ExportPayload, exportAllChats, importFromJson.

Importers: components/settings/SettingsModal.

lib/path.ts73 lines

Pure tree math — no Dexie, no React. buildTree(nodes) turns a flat Node[] into a forest of TreeNodes sorted by createdAt. layoutTree(roots) assigns x/y coordinates via a DFS where x increments per leaf. findPath(nodes, targetId) returns the ancestor IDs from root to target (inclusive).

Exports: TreeNode, LayoutPoint, buildTree, layoutTree, findPath.

Importers: components/chat/ChatApp (findPath), Sidebar (buildTree), Overlays (all three), pages/Chats (buildTree + layoutTree).

lib/markdown.tsx396 lines · lazy-loaded

Tiptap-based rich-text editor used for message editing. Imported via React.lazy from Message.tsx so the heavy ProseMirror + tiptap-markdown + markdown-it stack code-splits into its own async chunk. Configured with html: false so raw HTML is stripped at parse time.

Six sequence diagrams for the load-bearing paths.

Streaming a reply

Verified against useStream.send (src/hooks/useStream.ts lines 27–137) and streamMessage (src/lib/stream.ts lines 31–184). The user message is persisted to Dexie before the request fires, so when buildPathMessages walks the tree it picks the new message up and the model receives the actual current question.

Creating a branch

Three entry points: SelectionPopup → ChatApp.handleBranchFromSelection; per-message “branch from this” button → handleBranchFromMessage; composer “+ new branch” → handleCreateBlankBranch. All three land on createBranch, which is wrapped in an outer transaction so the branchMode === "stay" revert is part of the same committed write and live-queries don’t flicker.

File upload

The composer’s paperclip opens a hidden <input type="file" multiple>. Each file is classified by inferKind (image / pdf / code / file), converted to its storable form (base64 data URL for images via FileReader, pdf.js text extraction for PDFs, raw file.text() for code/text), then stored in db.files. Only a chip-sized ProcessedFile is kept in component state. On send, the file IDs ride along on the user message; at buildPathMessages time the contents are injected into the prompt (NOT into the textarea).

sequenceDiagram actor U as User participant C as Composer participant F as files.ts participant DB as Dexie participant CA as ChatApp participant H as useStream participant BPM as buildPathMessages participant OR as OpenRouter U->>C: pick files via paperclip C->>F: storeFile(file) F->>F: inferKind(file) alt image F->>F: FileReader.readAsDataURL else pdf F->>F: dynamic import("pdfjs-dist") + worker F->>F: getDocument + page.getTextContent else code/text F->>F: file.text() end F->>DB: db.files.add({content,name,kind,...}) F-->>C: ProcessedFile {fileId,name,kind,sizeBytes} C->>C: setFiles(prev => [...prev, stored]) Note over C: chip appears in files-row U->>C: send C->>CA: onSend({fileIds, composerText}) CA->>H: send(params) H->>DB: messages.add({fileIds, content, ...}) H->>BPM: buildPathMessages(chatId, nodeId) BPM->>DB: files.where("_id").anyOf(fileIds) BPM->>BPM: emit <document>/fence/image_url parts H->>OR: stream with parts in messages[]

Reasoning models

include_reasoning: true is set on every request body in stream.ts. delta.reasoning (or delta.reasoning_content from upstream providers) is routed to onReasoning; delta.content is routed to onChunk. useStream tracks them in separate state slots, and Stream / Message render the reasoning via the collapsible Reasoning component — auto-open while streaming, auto-collapse on completion, with the spinning four-pointed-star icon.

11 · openrouter request

The exact JSON body we POST.

Verbatim from src/lib/stream.ts lines 34–70.

URL + headers
POST https://openrouter.ai/api/v1/chat/completions
Content-Type:  application/json
Authorization: Bearer <cogninode_api_key>
HTTP-Referer:  https://github.com/rahulmohan/cogninode
X-Title:       cogninode beta
Body
{
  "model": "<openRouterId>",
  "messages": [
    { "role": "system", "content": "You are a helpful, expert AI assistant. ..." },
    ...pathMessages
  ],
  "stream": true,
  "stream_options": { "include_usage": true },
  "include_reasoning": true,
  "max_tokens": 16384
}

Field-by-field

FieldWhy
modelThe openRouterId from cost.ts (or a user-defined custom model). e.g. anthropic/claude-sonnet-4.5.
messagesSystem prompt is hard-coded in stream.ts. The remainder is whatever buildPathMessages returned for the current node; user messages with file attachments become structured image_url / <document> / fenced-code content.
streamSSE response.
stream_options.include_usageForces OpenRouter to emit a final usage object so we can compute costUsd client-side.
include_reasoningSurfaces delta.reasoning from reasoning models (DeepSeek R1, o1-style, Tencent HY3). Without this some providers swallow CoT and the user sees an empty reply that still cost tokens.
max_tokens16384. Was 4096 — that cap was too low for reasoning models that can spend the entire budget on CoT before any visible answer.

ChunkContentsWhen it loads
vendor React 18 + ReactDOM + react-router-dom Eager (parsed before first paint).
dexie Dexie 4 + dexie-react-hooks Eager. App boot imports db via useSettings…wait, actually via db.ts which is touched by both ApiKeyGate and every page; effectively eager.
streamdown streamdown (which pulls shiki + katex + remark/rehype/…) Loaded when MarkdownBody is first rendered. In practice that’s as soon as a chat opens. Kept in its own chunk so it can be cached across app updates.
async editor chunk (no manualChunks entry) lib/markdown.tsx — Tiptap + tiptap-markdown + ProseMirror Lazy. Message.tsx does React.lazy(() => import("../../lib/markdown")). Only ships when the user first opens an edit affordance.
pdfjs-dist (excluded from optimizeDeps) pdf.js + worker Lazy. files.ts calls await import("pdfjs-dist") only when processing a PDF.

optimizeDeps.exclude: ["pdfjs-dist"] stops Vite from pre-bundling pdf.js so the dynamic-import code path actually code-splits. assetsInclude: ["**/*.wasm"] covers the pdf.js worker’s static asset emission. The worker URL is captured at build time via import pdfWorkerUrl from "pdfjs-dist/build/pdf.worker.min.mjs?url" in files.ts so the worker version always matches the installed pdfjs-dist.
